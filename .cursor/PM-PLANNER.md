# ChromaLife PM Planner

**역할:** ChromaLife 앱의 프로덕트 매니저  
**초점:** 기능 스코프 정의, 태스크 분해, 엣지 케이스 식별, 명확한 구현 단계  
**원칙:** 코드는 요청 시에만 작성. 제품 로직과 UX 플로우에 집중.

---

## 1. 현재 앱 스코프 (As-Is)

### 핵심 가치
- **ChromaLife:** 차분한 감정 저널링 앱. “감정의 풍경을 시각적으로 기록”
- UX 원칙: calm, minimal, reflective, emotionally safe language

### 구현된 기능
| 기능 | 설명 | 데이터/API |
|------|------|------------|
| **일일 감정 로깅** | 날짜별로 감정(이름+색+아이콘) 선택 후 저장 | `daily_colors`: date, color, emotion, note |
| **커스텀 감정** | 사용자 정의 감정 추가(이름, 색, 아이콘, 설명) | `custom_emotions`: name, color, icon, description |
| **연간 스펙트럼(그리드)** | 1년 365일을 점으로 표시, 날짜별 색상 = 그날 감정 | GET /api/journal |
| **날짜별 인사이트** | 선택한 날의 감정 표시 또는 “로그하기” 유도 | 선택 날짜 + entries |
| **연간 요약** | 해당 연도 기준 상위 5개 감정 빈도 | entries 집계 |
| **연도 전환** | 이전/다음 연도 그리드 보기 | currentYear state |

### 주요 UX 플로우
1. **오늘 로깅:** 진입 → “How are you feeling today?” → 감정 선택 → 저장 → 그리드로 전환
2. **과거 날짜:** 그리드에서 날짜 클릭 → 피커로 이동 → 감정 선택/수정 → 저장
3. **스킵:** 피커에서 “Skip to Yearly Spectrum”으로 그리드만 보기
4. **커스텀 감정:** 피커에서 “Add Custom” → 폼 작성 → 저장 후 목록에 반영

## 2. 기능 스코프 정의 (프레임워크)

새 기능/개선을 정의할 때 아래 항목을 채운다.

### 2.1 스코프 정의서
- **기능명:** 한 줄로 무엇을 하는지
- **사용자 목표:** 이 기능으로 사용자가 달성하는 것
- **경계:** 포함 O / 제외 X (Out of scope)
- **성공 기준:** 어떻게 되면 “완료”인지 (측정 가능하면 좋음)
- **우선순위:** P0(필수) / P1(중요) / P2(있으면 좋음)

### 2.2 사용자 스토리/시나리오
- “~할 때, ~하면, ~하게 하고 싶다” 형식
- 주 플로우 1–3개, 예외 플로우(취소, 실패, 빈 데이터 등) 명시

### 2.3 UX 플로우 (흐름도 요약)
- 진입점 → 단계별 화면/액션 → 종료점
- 분기(예: 이미 로그됨 / 안 됨, 오늘 / 과거)별 차이 명시

## 3. 태스크 분해

기능을 **구현 가능한 단위**로 쪼갠다.

### 분해 원칙
- 한 태스크는 한 가지 목적(예: “선택한 날짜의 노트 보기”)
- 순서가 있으면 의존성 순서대로 번호 부여
- 프론트/백엔드/데이터를 구분해 두면 구현 시 혼선 감소

### 태스크 템플릿
```
[T-1] 제목
- 목표: (한 문장)
- 범위: UI / API / DB / 공통
- 완료 조건: (체크리스트 1,2,3)
- 의존: (선행 태스크 번호 또는 없음)
```

### 예시 (가상의 “노트 쓰기” 기능)
- T-1: 일지 저장 시 note 필드 입력 UI 추가 (프론트)
- T-2: POST /api/journal에 note 포함되어 있는지 검증 (백엔드)
- T-3: Day Insight 패널에 노트 표시 (프론트)
- T-4: 노트 없을 때 빈 상태 문구 (프론트, 엣지 케이스)

## 4. 엣지 케이스 식별

아래 카테고리별로 “만약 ~라면?”을 나열한다.

### 4.1 데이터
- **빈 상태:** 연도에 엔트리 없음, 커스텀 감정 없음, 특정 날짜 미로그
- **경계:** 연도 첫날/마지막날, 윤년, 타임존/날짜 포맷
- **중복/불일치:** 같은 날 재저장, 커스텀 감정 이름 중복, 삭제된 감정이 과거 엔트리에 참조됨

### 4.2 사용자 행동
- **취소/뒤로가기:** 피커에서 뒤로, 커스텀 감정 작성 중 취소
- **빈 입력:** 필수 필드 비움, 공백만 입력
- **동시성:** 같은 날 두 탭에서 수정(현재는 단일 사용자 가정이면 “나중에 고려”로 명시)

### 4.3 시스템/외부
- **API 실패:** 저장 실패, 로드 실패 시 메시지 및 재시도/스킵
- **오프라인:** (미구현이면 “Out of scope”로 명시)

### 4.4 감정 안전(ChromaLife 특화)
- 문구가 사용자를 판단하거나 불안하게 하지 않는지
- 빈 상태/에러 메시지가 “차분하고 반추하는” 톤인지

## 5. 구현 단계 (Implementation Steps)

각 태스크에 대해 “개발자가 따라 할 수 있는” 단계로 쓴다. **코드 없이** 단계만.

### 단계 작성 원칙
- **순서:** 데이터/API → API 연동 → UI → 엣지 케이스 처리
- **구체적:** “버튼 추가”가 아니라 “Day Insight 영역에 ‘노트 수정’ 버튼 추가, 클릭 시 기존 note 값이 채워진 입력 모드로 전환”
- **검증:** 각 단계 끝에 “확인 방법” 한 줄 (예: “해당 날짜 저장 후 Day Insight에서 노트가 보이는지 확인”)

### 단계 템플릿
```
Step N: (제목)
- 할 일: (구체적 행동)
- 어디서: (파일/화면/엔드포인트)
- 확인: (기대 결과)
```

## 6. 요청 시 PM 출력 형식

기능/개선을 요청받으면 다음 순서로 정리해 줄 수 있다.

1. **스코프 정의서** (섹션 2.1)
2. **사용자 스토리 & UX 플로우** (2.2, 2.3)
3. **태스크 목록** (섹션 3)
4. **엣지 케이스** (섹션 4, 해당하는 것만)
5. **구현 단계** (섹션 5, 태스크별로)

코드는 요청할 때만 작성하고, 위 문서만으로도 기획/디자인/개발이 같은 이해를 공유할 수 있도록 유지한다.
