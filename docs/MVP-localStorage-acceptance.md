# ChromaLife MVP: localStorage 기반 하루 1개 기록

**버전:** MVP  
**저장소:** localStorage (서버/DB 없음)  
**제약:** 하루에 **1개 엔트리**만 유지 (키 = 오늘 날짜)

---

## 1. 스코프 정의서

| 항목 | 내용 |
|------|------|
| **기능명** | 오늘 날짜 기준 엔트리 1개 불러오기 / 저장 / 수정 (localStorage) |
| **사용자 목표** | 오늘의 감정을 한 번에 하나만 기록하고, 필요 시 수정할 수 있다. |
| **In scope** | 오늘 엔트리 로드, 최초 저장, 기존 엔트리 수정, localStorage 읽기/쓰기 |
| **Out of scope** | 과거 날짜 선택·수정, 연간 그리드, 커스텀 감정, 서버 API, 다중 엔트리 |
| **성공 기준** | 새로고침 후에도 오늘 엔트리가 유지되고, 수정 시 덮어쓰기된다. |
| **우선순위** | P0 (MVP 필수) |

---

## 2. 오늘 엔트리 — Acceptance Criteria

### 2.1 오늘 엔트리 불러오기 (Load)

| ID | 조건 | 기대 결과 |
|----|------|-----------|
| **L1** | 앱 진입 시 localStorage에 **오늘 날짜 키**에 해당하는 엔트리가 **있음** | 해당 엔트리(date, color, emotion, note 등)를 화면에 표시한다. |
| **L2** | 앱 진입 시 localStorage에 **오늘 날짜 키**에 해당하는 엔트리가 **없음** | “오늘 기록이 없음” 상태로 표시하고, 감정 선택/저장 플로우를 유도한다. |
| **L3** | 오늘 날짜 키는 **로컬 날짜 기준**으로 고정 (예: `YYYY-MM-DD`) | 자정이 지나면 “오늘”이 바뀌고, 다음 날은 새 키로 빈 상태가 된다. |
| **L4** | 저장된 데이터 구조가 **깨졌거나** 필수 필드가 없음 | 깨진 데이터는 사용하지 않고, 빈 상태처럼 취급한다. (에러로 크래시하지 않음) |

### 2.2 오늘 엔트리 저장 (Save)

| ID | 조건 | 기대 결과 |
|----|------|-----------|
| **S1** | 오늘 **처음** 감정을 선택하고 저장함 | localStorage에 오늘 날짜 키로 엔트리가 생성되고, 화면에 저장된 내용이 반영된다. |
| **S2** | 이미 오늘 엔트리가 있는 상태에서 **다른 감정**을 선택하고 저장함 | 기존 오늘 엔트리를 **덮어쓰기**하고, 화면은 새 감정으로 갱신된다. (하루 1개 유지) |
| **S3** | 저장 직후 **새로고침**함 | 새로고침 후에도 방금 저장한 오늘 엔트리가 그대로 보인다. |
| **S4** | 저장할 엔트리에 **필수 필드**(날짜, 감정, 색 등 정의된 최소 필드)가 모두 있음 | 정상 저장된다. |
| **S5** | 저장 시 **필수 필드가 비어 있음** (예: 감정 미선택) | 저장 버튼 비활성화 또는 저장 시 검증 후 저장하지 않고, 사용자에게 입력 유도 메시지를 보여준다. |

### 2.3 오늘 엔트리 수정 (Edit)

| ID | 조건 | 기대 결과 |
|----|------|-----------|
| **E1** | 오늘 엔트리가 **이미 있는** 상태에서 “수정” 또는 감정 재선택 후 저장함 | 기존 오늘 엔트리가 새 값으로 **업데이트**되고, 화면에 즉시 반영된다. |
| **E2** | 수정 후 **새로고침**함 | 수정된 내용이 유지된다. |
| **E3** | 오늘 엔트리가 **없는** 상태에서는 “수정”이 아니라 “기록하기” 플로우만 노출 | 수정 전용 UI는 빈 상태에서 보이지 않거나, “기록하기”와 동일한 플로우로 진입한다. |

## 3. 엣지 케이스 정리

### 3.1 날짜·시간

| 케이스 | 설명 | 처리 방안 |
|--------|------|-----------|
| **자정 경계** | 사용자가 23:59에 저장하고 00:01에 다시 열면 “오늘”이 바뀜 | 새 날짜는 빈 상태. 이전 날 엔트리는 다른 날짜 키에 있으므로 MVP에서는 “오늘만” 보므로 그대로 두어도 됨. (과거 조회는 Out of scope) |
| **타임존** | 사용자 기기 로컬 날짜 사용 | `new Date()` 로컬 날짜 기준 `YYYY-MM-DD` 생성. 서버 없으므로 타임존 동기화 이슈 없음. |
| **날짜 키 포맷** | 일관된 키 포맷 필요 | 한 가지 포맷 고정 (예: `chromalife:YYYY-MM-DD`). 파싱 실패 시 해당 키 무시. |

### 3.2 데이터 무결성

| 케이스 | 설명 | 처리 방안 |
|--------|------|-----------|
| **localStorage 비어 있음** | 해당 키에 데이터 없음 | L2와 동일 — 빈 상태 UI. |
| **손상된 JSON** | 문자열이 JSON으로 파싱되지 않음 | try/catch 후 실패 시 빈 상태로 간주, 크래시 방지. |
| **구 스키마** | 이전에 다른 키/형식으로 저장된 데이터 | MVP에서는 “오늘” 키만 읽음. 구 키는 무시해도 됨. (마이그레이션은 이후 검토) |
| **필수 필드 누락** | emotion, color 등 필드 없음 | 유효성 검사 후 불완전하면 “저장 안 됨”으로 간주하고 빈 상태처럼 표시 또는 기본값으로 보정 (정책 결정). |

### 3.3 저장 실패·제한

| 케이스 | 설명 | 처리 방안 |
|--------|------|-----------|
| **localStorage 풀** | 브라우저 저장 한도 초과 (드묾) | 저장 실패 시 사용자에게 “저장 공간이 부족할 수 있습니다” 등 차분한 안내. 재시도 가능. |
| **시크릿/비공개 모드** | 일부 환경에서 localStorage 비활성화 | 읽기/쓰기 실패 시 “이 환경에서는 기록이 저장되지 않을 수 있어요” 등 톤 유지. |
| **저장 중 탭 종료** | 저장 직후 사용자가 탭을 닫음 | 동기 API라 보통 저장 완료됨. 명시적 “저장됨” 피드백이 있으면 UX 개선. |

### 3.4 사용자 행동

| 케이스 | 설명 | 처리 방안 |
|--------|------|-----------|
| **감정 선택 없이 저장 시도** | 저장 버튼만 누름 | S5 — 버튼 비활성화 또는 클릭 시 “오늘의 감정을 골라 주세요” 등 유도. |
| **같은 감정 다시 선택 후 저장** | 이미 저장된 감정과 동일하게 선택 후 저장 | 허용. 덮어쓰기로 처리 (동작 일관성). |
| **빈 note** | note는 선택 사항이라면 | 빈 문자열 또는 null 저장 허용. 표시 시 “메모 없음” 등으로 처리. |

### 3.5 감정 안전 (ChromaLife 톤)

| 케이스 | 설명 | 처리 방안 |
|--------|------|-----------|
| **빈 상태 문구** | “기록이 없음” 표현 | 판단하지 않는 표현 (예: “오늘의 색을 남겨 보세요”, “아직 기록이 없어요”). |
| **저장 실패 메시지** | localStorage 실패 시 | “지금은 저장이 되지 않았어요. 잠시 후 다시 시도해 주세요.” 등 차분한 톤. |
| **수정 유도** | 기존 엔트리 있을 때 | “바꾸고 싶다면 다시 골라 주세요” 등 부담 없이 수정 가능함을 전달. |

---

## 4. 요약 체크리스트 (구현 후 검증용)

- [ ] **불러오기:** 오늘 키 있음 → 표시 / 없음 → 빈 상태. 깨진 데이터 시 빈 상태.
- [ ] **저장:** 최초 저장 시 생성, 이후 저장 시 덮어쓰기. 새로고침 후 유지.
- [ ] **수정:** 오늘 엔트리 있을 때만 수정 플로우, 저장 시 업데이트 반영.
- [ ] **날짜:** 로컬 날짜 기준 `YYYY-MM-DD` 키, 자정 넘기면 새 날짜 = 빈 상태.
- [ ] **엣지:** JSON 손상·localStorage 실패·필수 필드 누락 시 크래시 없이 처리, 메시지 톤 유지.
